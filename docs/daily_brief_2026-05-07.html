一、Agentic Engineering：AI 编程的瓶颈终于从“会不会写”滚回了“敢不敢交付”

今天最硬的信号来自 Simon Willison 对 Code w/ Claude 2026 的现场记录，以及他同日写下的 “Vibe coding and agentic engineering are getting closer than I’d like”。这不是又一轮“AI 要替代程序员”的廉价烟花，而是一个更残酷的工程事实：模型写代码这件事已经不稀缺，稀缺的是规格、边界、验证、审计、回滚、责任链。以前大家争论 copilot 到底是补全器还是实习生，现在问题升级成：当一个 agent 能读 repo、改多文件、跑测试、开 PR、解释 diff，人类到底还在控制什么？

背景要往回看。vibe coding 最初像创业者的肾上腺素玩具：一句话丢给模型，生成一个能演示的 app，截图发 X，融资 PPT 里贴“10x builder”。它适合原型，适合低风险工具，适合把想法从脑子里拽到屏幕上。但专业工程的债不是这么算的。线上系统要面对权限、数据迁移、兼容性、支付回调、异常恢复、供应链漏洞、测试覆盖、审计证据。Agentic engineering 的出现，本质上是把“AI 写代码”重新塞回工程纪律里：需求要拆，约束要明，工具权限要小，结果要验，人类要审。

Simon 的新感悟锋利在这里：他原本用 agentic engineering 区分负责任的 AI 辅助开发和放飞自我的 vibe coding，但两者正在靠近。因为工具越来越强，越强越诱人，人类越容易把“我看过大概没问题”伪装成“我审计过”。这就是最危险的地方。企业买 coding agent，不是买一个更会打字的模型，而是在买一套“谁能动代码、谁能触发命令、谁能接触密钥、谁对产物签字”的生产关系。Anthropic、OpenAI、Replit 这类玩家会把产品从聊天窗口推进工程流程编排层，真正的钱不在 token，而在团队把 agent 纳入研发制度后愿意按席位、按项目、按产出持续付费。

生产环境配置演示：把 agent 当成受限施工队，不要当成许愿池

agent_policy:
workspace: services/payments
objective: implement_invoice_retry_policy
allowed_paths:
- services/payments/src
- services/payments/tests
- docs/payments
forbidden_paths:
- infra/secrets
- migrations/production
- vendor
allowed_commands:
- npm test -- --runInBand services/payments
- npm run lint -- services/payments
- npm run typecheck -- services/payments
require_before_edit:
- read_architecture_notes
- list_public_interfaces
- identify_rollback_plan
require_before_merge:
- human_review
- passing_tests
- security_diff_scan
- changelog_entry
output_contract:
- changed_files
- behavior_delta
- verification_evidence
- unresolved_risks

这个配置的核心不是漂亮，而是把 agent 从“随便跑命令的天才儿童”降级成“带工牌的外包工程队”。它可以写，但只能在指定目录写；它可以跑命令，但只能跑验证命令；它可以解释，但必须输出风险和证据。你真正需要的是这样的窄门：每一次自动改代码都留下可追踪痕迹，每一次通过都能被复盘，每一次失败都能回滚。

实战优劣势分析：
优势一，交付速度确实会抬升。对于熟悉 repo 的工程师，agent 能把重复性改动、样板测试、接口适配、文档同步一扫而空。优势二，知识库接入后，新人 onboarding 的一部分会被压缩，尤其是“这个模块为什么这么写”的历史包袱。优势三，小团队可以同时推进更多低风险变更，过去排队等人的活会被拆给多个 agent 并行处理。

劣势也更致命。第一，agent 最擅长制造“看上去合理”的错误，尤其在边界条件和跨服务契约上，错得很像高级工程师写的。第二，验证会变成新瓶颈。代码生成便宜了，测试设计、评审、灰度、监控反而更贵。第三，权限一旦给大，供应链事故会被放大；agent 不是恶意的，但它会忠实执行一个含糊目标，含糊目标在生产环境里就是债券违约。第四，团队会出现新的绩效幻觉：PR 数量暴涨，真实质量不一定涨，管理层若只盯活动指标，等于拿 AI 给假勤奋上杠杆。

Dolan's 锐评：
AI 编程的下一轮胜负，不看谁的 demo 更像魔法，看谁敢把 agent 塞进审计、权限、测试和回滚里还不炸。Simon 这波提醒很值钱：vibe coding 不是原罪，装作自己在工程化才是。最可笑的是一堆公司嘴上喊“开发者被替代”，实际采购表里第一项就是“人类审批在哪里”。钱往哪流？流向能把模型关进笼子的工具，而不是给模型戴王冠的 PPT。

二、模型实验室开始“卖服务”：API 毛利神话往下沉，AI 公司要亲自抢咨询商饭碗

Latent Space 的 AINews 把这轮动作概括成 “Silicon Valley gets Serious about Services”。这句话比一堆发布会标题都诚实：模型实验室不满足于卖 API，不满足于开发者自己来调，不满足于平台在旁边收过路费，它们要往最后一公里伸手。Agent lab 叠加 services company，听起来像产业升级，翻译成人话就是：模型能力再强，客户的业务系统烂、权限系统乱、数据字典脏、流程全靠 Excel 和口口相传，模型自己进不去。于是实验室只能下场做集成、流程改造、岗位重排和成果交付。

这背后是商业模式的硬拐点。API 调用量是一门漂亮但残酷的生意：价格持续下探，模型同质化，客户随时换路由，毛利被推理成本和降价竞争夹住。SaaS 席位也不轻松，因为 AI 不是简单给每个员工加一个按钮就能变钱。真正的大市场在服务业和知识工作流程：金融分析、法务尽调、客服处理、医疗文书、研发管理、合规审查、销售运营。问题是这些场景不是单点功能，而是“业务流程加数据权限加组织责任”的混合物。谁能把模型接到流程里，谁才拿得到预算。

Latent Space 提到的关键线索是 Anthropic 与资本方围绕企业服务的动作，以及 OpenAI 相关部署公司的资本叙事。无论具体结构如何，方向已经很清楚：模型实验室要从“卖铲子”变成“带施工队承包金矿”。这对 Accenture、Deloitte、McKinsey 这类传统服务商不是马上死刑，但等于头顶多了一个会写代码、懂模型、握 API 定价权、还能拿私募钱补贴交付的竞争者。传统咨询最值钱的是行业关系和变革管理，模型实验室最值钱的是底层能力和产品速度。双方迟早会在企业 IT 的预算池里互相撕。

这才像一个能卖给企业的 Agent 服务。它不是“帮我总结一下合同”的玩具，而是明确数据输入、工具边界、审批点、审计日志、SLA 和收费方式。模型实验室想拿服务收入，就必须承认一个难听事实：企业买的不是 intelligence，而是可控的业务结果。没有日志，法务不会签；没有权限隔离，CISO 会掀桌；没有人类审批，业务负责人不敢背锅；没有 SLA，采购只会把你当实验项目。

实战优劣势分析：
优势是钱厚。服务预算比 API 预算更接近老板的疼点，能直接对应降本、提速、减少人力、提高合规质量。优势是数据更深。做服务交付能拿到真实流程反馈，这比公开 benchmark 更能训练产品判断。优势是护城河更硬。一个嵌入客户工作流、绑定权限和审计的 Agent，比一个普通模型 API 更难被替换。

劣势同样扎心。第一，服务是脏活累活，毛利不一定像软件那么性感。客户系统千奇百怪，交付团队会被定制需求拖进泥地。第二，责任会变重。API 时代可以说“用户自行使用”，服务时代你替客户执行流程，出了事就别装路人。第三，组织能力要求完全不同。模型研究员擅长推 frontier，服务交付需要项目经理、行业专家、解决方案架构师和合规人员，这不是多招几个 prompt engineer 就能糊弄。第四，传统服务商不会坐着等死，它们有客户关系，有采购通道，有本地团队，一旦绑定头部模型，也会反杀一波。

Dolan's 锐评：
AI 行业以前最爱讲“软件吞噬世界”，现在轮到模型公司自己吞服务业，结果发现服务业不是薯片，是带刺的河豚。API 是卖铲子，Agent 服务是下矿井，利润更大，事故也更真实。谁能把模型、流程、权限、变革管理揉成一台能赚钱的机器，谁吃肉；谁还在官网放三张聊天截图，谁就是给资本市场演脱口秀。

三、临床 LLM 的安全性与准确率不同步：医疗 AI 的参数崇拜被论文当场拆台

arXiv 论文 “Safety and accuracy follow different scaling laws in clinical large language models” 给医疗 AI 浇了一盆冷水。它的核心提醒很简单但杀伤力巨大：临床大模型的 accuracy 和 safety 不一定随着规模同步提升。也就是说，模型在医学问答 benchmark 上更会答题，不代表它在真实临床场景里更安全。医疗 AI 公司最爱拿准确率曲线当融资护身符，这篇论文等于把桌子掀开：会做题和不会害人，是两条不同曲线。

背景不复杂。过去两年，医疗 LLM 的宣传套路高度一致：模型参数更大、医学考试分数更高、文献理解更强、医生节省时间更多。问题是临床不是选择题赛场。真实医疗里，安全包含拒答、识别高风险症状、提示就医、避免过度自信、处理信息缺失、尊重本地指南、记录证据链、区分辅助建议与诊断决策。一个模型可能非常擅长给出看似完整的解释，却在关键禁忌、药物相互作用、急症识别上翻车。医疗场景容错率低，错误不是“体验不好”，而是可能把人送进 ICU。

这篇论文的重要性在于它把监管、投资和产品设计的焦点拉回安全评估。医疗 AI 不能再靠单一综合分遮羞。评测要拆成多轴：医学事实准确性、风险识别、伤害避免、拒答质量、解释透明度、临床流程适配、上市后监测。模型变大带来的收益可能在某些轴上递增，在另一些轴上停滞甚至反向。若企业只展示 accuracy，就是把风险藏进营销材料里，等着医院、保险公司和监管来收尸。

生产环境评测配置演示：医疗 LLM 上线前别问“准不准”，先问“哪里会害人”

clinical_llm_eval:
model: hospital_assistant_candidate
datasets:
- medical_qa_accuracy
- emergency_triage_red_flags
- medication_interaction_safety
- hallucinated_citation_detection
- refusal_and_escalation_cases
- local_guideline_alignment

- all_patient_facing_answers_logged

monitoring:

这套配置不性感，但它像医疗。它承认模型不是医生，而是临床流程里的受控工具。先 shadow mode，不直接面对患者；再医生审核；最后才有限开放。每个高风险回答都要能追溯，严重事件触发回滚。医疗 AI 真要进入医院，靠的不是模型演示多会说人话，而是出了问题时能不能找到证据、责任和刹车。

实战优劣势分析：
优势是方向正确。医疗文书、检索、随访、质控、医保控费都有刚需，LLM 可以减轻医生大量低价值文本劳动。优势是安全评估一旦做扎实，会形成竞争壁垒，后来者很难靠一个更大模型直接替换。优势是本土化空间大，中国的指南、药品、医保政策、医院流程都需要本地评测集。

劣势是落地慢。医疗采购谨慎，数据出院难，医生信任建立慢，监管证据要求高。另一个劣势是 ROI 不像客服机器人那么直观，节省多少时间、减少多少事故、谁来付费，都要算细账。最致命的是营销冲动：一旦公司为了融资把辅助诊疗吹成“AI 医生”，就会把自己从工具赛道吹进高责任雷区。

Dolan's 锐评：
医疗 AI 最怕的不是模型不够聪明，而是模型聪明到会把错误说得像主任医师。准确率是舞台灯，安全性才是手术灯。投资人若还只看榜单分数，就是拿病人的风险给估值上色。未来能活下来的医疗 LLM 公司，不是参数最大那家，而是最会承认“不知道”、最会升级给真人、最会留下证据那家。
