■ Claude Code Computer Use
https://x.com/claudeai/status/2038663014...

2026年3月30日，Anthropic 官方账号 @claudeai 宣布 Claude Code 新功能 Computer Use 上线。Claude 可通过 CLI 直接打开本地应用、点击 UI 元素、执行构建物测试，实现端到端开发流程闭环。当前为 Research Preview 阶段，仅向 Pro 和 Max 计划用户开放。这是继 OpenAI Operator、OpenClaw 之后，AI Agent 领域最具野心的 GUI 操控尝试——也是最具争议的一次。

Computer Use 的技术原理并不神秘：本机安装 Claude Code 客户端后，Claude 通过截取屏幕区域（screenshot）感知 GUI 状态，结合 OS X Accessibility API 模拟鼠标点击和键盘输入。这套"看屏幕→做动作"的循环，本质上是多模态大模型的视觉输入输出在 GUI 场景的工程落地。但它的战略意义远比技术细节更值得深挖。 第一层是"最后一公里"问题。传统 coding agent 的局限在于：能生成代码，但无法验证代码在真实 GUI 环境下是否 work。Unit test 可以跑通，但"用户点击提交按钮后弹窗是否正常"这类问题，terminal 层的 agent 根本无法触及。Computer Use 把这个 gap 填上了——Claude 现在不仅能写代码，还能坐在你的工位上帮你点掉那些烦人的 cookie banner。 第二层是"代理层战争"的升维。OpenAI 的 Operator 走的是浏览器插件路线，专注于 web 自动化；OpenClaw 在终端层做了深度集成。Anthropic 则选择了操作系统级集成——直接在 macOS 上跑 GUI 测试。这是一个更激进的路线：它要求用户授予 Accessibility 权限（这在 macOS 里是受保护的能力），意味着只有信任 agent 的用户才会开启这个功能。这不是技术问题，是信任问题，而 Anthropic 选择用 Research Preview 来慢慢建立这种信任。 第三层是安全边界的新挑战。传统沙箱防不住一个拥有屏幕读取+鼠标操控权限的 AI Agent——它理论上可以帮你点"同意"，也可以帮自己授权更多权限。Anthropic 在 Computer Use 上的安全设计（截断的 screenshot、多层确认机制）是工程层面的妥协，但真正的安全范式还没建立。这为 red team 研究者提供了一个新的攻击面，也为一个新行业——AI Agent 安全审计——埋下了伏笔。

：
安装 Anthropic 官方 CLI 并启用 Computer Use（Research Preview 模式）：
npm install -g @anthropic-ai/claude-code
claude-code computer-use enable
claude-code --computer-use

测试任务示例（自动打开 Safari 并搜索关键词）：
claude-code << 'EOF'
Open Safari and search for "Anthropic Claude Computer Use"
Click on the first result
EOF

在 Claude Code 配置文件（~/.claudeirc.json）中启用截图精度设置：
{
  "computer_use": {
    "screenshot_mode": "retina",
    "click_precision": "high",
    "max_retries": 3,
    "preview_only": true
  }
}

■ Claude Cowork GA vs OpenClaw

2026年3月17-18日，Anthropic 发布 Claude Cowork 正式版（GA），并宣布支持远程控制功能。Claude Cowork 是 Anthropic 继 Claude Code Desktop 之后推出的多 Agent 协作产品，允许用户同时运行多个 Claude 实例协同工作。更重要的是：这是 Anthropic 对 OpenClaw（NVIDIA 在 GTC 上高调宣布的产品）的直接回应。战场从"谁有最强模型"转移到了"谁有最好的 Agent 部署平台"。

理解 Claude Cowork 的战略价值，需要回到一个被低估的信号：2026年1月，Anthropic 的 Felix Rieseberg 发推透露，非技术用户正在用 Claude Code 做各种"非编码工作"，而技术用户则在用它处理非技术任务。这条推文获得了 1.72K likes——说明这个现象是广泛共识，而非个例。 这个观察直接催生了 Claude Cowork。它本质上是一个多实例 Agent 编排层：用户不再是和单个 Claude 对话，而是同时调度多个 Claude 实例，每个实例负责不同的子任务。以"准备一份融资材料"为例：一个 Claude 负责收集竞品数据，一个负责写财务模型，一个负责设计 PPT——三者之间通过共享上下文协调，最终由人类汇总。Felix 透露，这个产品"在一周半内自己写好了自己"，完全由多个 Claude Code 实例协作完成开发。 这就是"Agentic Engineering"的工程化落地——不再是单 agent 的 prompt 工程，而是多 agent 的 workflow 编排。Simon Willison 在 Pragmatic Summit 的炉边对话中把这个问题讲得更清楚：AI coding adoption 分四个阶段，从 ChatGPT 问答，到 coding agent 写代码，到 agentic engineering 多 agent 协作，最后是 AI 原生（vibe coding，非程序员主导）。Claude Cowork 把第三阶段直接产品化了。 现在来看 OpenClaw——NVIDIA 在 GTC 2026 上高调宣布的产品，Jensen 在 keynote 里用了大量篇幅介绍它与 NVIDIA 整个生态（CUDA、TensorRT、NeMo）的深度集成。这里出现了一个有趣的分叉：Anthropic 的 Cowork 走的是"轻量协作层"路线，易用性强；OpenClaw 走的是"深度硬件绑定"路线，性能最强但迁移成本高。两者的竞争，本质上是"AI native 的 macOS"和"AI native 的 Linux"之间的路线之争。 更深层的问题是：谁来控制 Agent 的执行层？Anthropic 的答案是"人类的 hypervisor"（Cowork），NVIDIA 的答案是"NVIDIA 的 CUDA 层"（OpenClaw）。前者更灵活但护城河浅，后者更强大但生态锁定深。这场竞争的结果，将决定未来 5 年 AI Agent 的基础设施标准。
