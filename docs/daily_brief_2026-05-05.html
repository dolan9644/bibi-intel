一、医疗 RAG 和医保广告像素同时爆雷：患者入口正在变成数据绞肉机

背景溯源：今天最刺眼的不是又一个模型跑分，也不是谁把语音延迟压低了几十毫秒，而是医疗数据链路两头同时漏风。TechCrunch 报道美国医保 marketplace 向广告技术平台共享公民身份、种族等敏感注册数据，HN 把它顶到高热，说明工程社区已经不把这事当普通隐私新闻，而是当基础设施事故看。另一边，arXiv 医疗 AI 案例研究指出面向患者的 RAG 聊天机器人可能暴露后端、隐私和安全边界。这两个信号拼在一起，画面非常难看：前端是医保注册表单、广告归因像素、CRM 标签；后端是 RAG 检索、日志、提示注入、PHI 数据池。中间只要一个增长团队喊一句“转化率要归因”，一个产品经理喊一句“患者问答要智能化”，患者最脆弱的健康场景就被包装成了数据矿。

完整分析：医疗 AI 最大的幻觉，是以为风险主要来自“模型回答错”。错。回答错当然会害人，但更根本的风险是系统把不该暴露的东西交给了不该访问的人，甚至交给广告网络。RAG 系统一旦接入病历、保险、药品、预约、支付、客服工单，它就不是聊天机器人，而是一个挂着温柔 UI 的内部数据查询层。患者输入“我这个病能不能报销”，系统可能触发检索、拼接上下文、写日志、调用外部观测平台、把异常上报到 SaaS。任何一个环节没有字段级最小化，都会把医疗隐私变成增长漏斗里的调味料。

医保 marketplace 的广告数据问题更像传统互联网的老病复发：像素、归因、再营销、lookalike audience，都是广告业卖铲子的旧套路。区别只是这次矿不是鞋码和浏览记录，而是公民身份、种族、医保注册状态。监管层面，HIPAA 并不能覆盖所有 marketplace、广告平台和州级数据流，FTC 的 unfair/deceptive practices、州隐私法、敏感属性处理规则都会一起杀过来。对中国市场也别装没看见：互联网医院、药企患者服务、小程序埋点、保险导流、健康问卷、私域 CRM，本质上是同一条链。今天美国被扒，明天就是国内某个“AI 健康助手”被问：你的日志里到底有什么。

服务端还必须做对象级授权，不能把 patient_id 当成前端传来的普通参数。最小可用检查如下：

function authorizeRetrieval(user, request) {
if (!user || !user.sessionVerified) return { scope: "public" };
if (request.dataClass === "phi" && !user.stepUpVerified) throw new Error("step-up auth required");
if (request.patientId !== user.patientId) throw new Error("cross-patient access denied");
return { scope: request.dataClass, patientId: user.patientId };
}

function sanitizeForLogs(payload) {
return redact(payload, {

大佬们的新感悟：这组信号里最有价值的“大佬”不是机场演讲台上的模型布道者，而是一线 practitioner 和研究者。TechCrunch 原文把医保广告链路的敏感字段问题摆上桌面，链接：https://techcrunch.com/2026/05/04/us-healthcare-marketplaces-shared-citizenship-and-race-data-with-ad-tech-giants/ 。arXiv 医疗 RAG 安全案例把 patient-facing chatbot 的后端暴露风险拆出来，链接：https://arxiv.org/abs/2605.00796v1 。HN 讨论的价值在于它把媒体报道转成工程问题：像素、日志、权限、检索、合规审计，每一项都不是 PPT 上的“安全设计”，而是上线后会被 subpoena、监管、红队和记者轮流拷打的证据链。

实战优劣势分析：医疗 RAG 的优势是真实存在的。它能把保险条款、用药说明、预约流程、复诊建议从客服泥潭里捞出来，降低患者的信息摩擦，也能把医生和客服从重复问答里解放一部分。但它的劣势更要命：数据高度敏感、错误代价高、系统链路长、供应商多、日志和观测工具默认喜欢“全量吃掉”。广告归因也不是完全无价值，公共项目确实需要知道用户从哪里来、流程卡在哪里。但一旦把敏感字段塞进广告网络，增长团队省下来的 CAC 会变成法务和公关的火葬场账单。

Dolan's 锐评：医疗 AI 现在最大的泡沫，不是模型不够神，而是每家公司都想一边穿白大褂，一边拿广告漏斗当收银机。患者不是 DAU，病历不是增长素材，RAG 也不是给内部数据库开后门的高级皮肤。谁还把“先上线再补合规”当互联网方法论，谁就是在医疗赛道给自己买监管期权，而且是深度虚值、随时爆仓那种。

二、Coding Agent 从炫技走向可复现：科学工作流、格式化迁移和安全约束正在重写协作框架

背景溯源：今天的 Agent 主线不在“又能写几行代码”，而在“写出来的东西能不能被组织、科研和安全系统承认”。arXiv 研究 coding agents 能否复现实验性计算材料科学发现，直接把代码代理从 LeetCode 玩具场拉到科研复现现场。Stripe 工程博客讲 rubyfmt 在 2500 万行 Ruby 代码库中一夜完成格式化迁移，展示的是大规模机械变更如何被组织化、审计化、回滚化。Strix 披露 DoD 背景创业公司的多租户授权漏洞，又提醒所有企业 Agent 平台：workspace、org、project、connector 这些权限边界不是 UI 里的层级标签，而是能不能被越权打穿的钢筋。

完整分析：过去一年，coding agent 的卖点集中在“能替你写代码”。这话太浅，像金融销售只会喊收益率。真正的问题是：它能不能复现复杂实验？能不能在大型代码库里做低语义风险的机械变更？能不能在多租户 SaaS 里理解权限和租户边界？能不能把每一步改动留下可审计证据？如果答案是否，所谓 agent 就只是一个会生成 diff 的实习生，还要 senior engineer 给它擦屁股。

科研自动化尤其残酷。材料科学发现不是普通 CRUD，里面有实验参数、数值稳定性、库版本、随机种子、数据清洗、论文描述和隐性领域知识。Coding agent 要复现这类发现，必须同时具备代码阅读、环境重建、实验编排、错误诊断和结果解释能力。它的价值不是“写得快”，而是把不可复现的科研叙事压成可执行工作流。资本市场为什么盯 AI scientist？因为一旦复现、探索、仿真和实验闭环被 agent 接管，科研效率的边际成本会下台阶。但这里也有巨大泡沫：实验室不是演示视频，失败结果、脏数据、设备漂移、论文省略步骤，都会把 agent 的漂亮推理按在地上摩擦。

Stripe 的 rubyfmt 案例则是企业工程的另一面。2500 万行代码一夜格式化，真正难的不是 formatter，而是变更窗口、冲突控制、CI 性能、review 噪音、回滚策略、团队沟通。对 Codex、Claude Code、OpenClaw 这类协作系统而言，这说明 agent 不能只当局部编辑器插件，它要理解组织级变更的节奏：什么时候适合大规模机械 diff，什么时候必须拆小 PR，什么时候需要冻结分支，什么时候需要生成迁移说明。格式化迁移是最温和的练兵场，后面还有 API 替换、安全修复、依赖升级、架构约束迁移，才是真正的钱往哪流。

生产配置演示：一个企业级 coding agent 平台最少要有任务权限、租户隔离、变更审计、验证门禁。下面是一份可落地的 agent policy，不是摆设。

权限检查不要靠路由装样子，要在数据访问层重复执行。示例：

function requireTenantAccess(session, object) {
const tenantId = session.claims.tenant_id;
if (!tenantId) throw new Error("missing tenant claim");
if (object.tenant_id !== tenantId) throw new Error("tenant isolation violation");
return true;
}

async function loadProject(session, projectId) {
const project = await db.projects.findById(projectId);
requireTenantAccess(session, project);
await audit.write({ actor: session.agent_id, action: "project.read", projectId, tenant: session.claims.tenant_id });
return project;
}

实战优劣势分析：Coding agent 的优势非常直接：处理重复修改、生成测试、补文档、迁移 API、复现实验环境，能把 senior engineer 从大量低创造性劳动里捞出来。它在科学工作流里还有额外价值：把论文里的隐性步骤显性化，把环境和结果记录下来，把失败实验也结构化保存。劣势同样硬：agent 天生会自信地越权、自信地改错、自信地把上下文里的秘密写进日志。多租户平台一旦边界设计松，agent 不是生产力工具，而是自动化越权放大器。大型机械变更如果没有组织窗口，也会把团队的 merge queue 搅成浆糊。

Dolan's 锐评：Agent 的下一轮竞争不会是谁更会“像人一样写代码”，而是谁更会像一家严肃公司一样留下证据、遵守权限、接受审计、失败可回滚。卖 demo 的厂商还在喊“一个人顶十个人”，真正在掏钱的企业只问三件事：会不会泄密，会不会跨租户，会不会把 main branch 炸穿。答不上来，就别装未来同事，你只是一个昂贵的自动补全。
