■ GitHub 4月24日大限
https://github.com/settings/pages... · https://news.ycombinator.com/item?id=475... · https://github.blog/changelog/2026-03-ai...

微软GitHub于2026年3月正式宣布，若用户不在4月24日前通过设置禁用，GitHub将默认授权其所有代码——包括私有仓库——用于训练Copilot等AI模型。HN讨论帖瞬间爆了298 points、130条评论。开源社区炸锅，白帽社区愤怒，隐私倡导者发出最高级别警报。

这次事件远非普通隐私条款更新，而是一场精心设计的"沉默同意"博弈。GitHub选择了Opt-Out而非Opt-In机制——也就是说，全球数以亿计的开发者，除非主动去设置页打勾，将被默认纳入AI训练数据池。这不是bug，这是feature。Opt-Out机制在法律上完全合规，在道德上却是一场豪赌——赌的是绝大多数用户永远不会去改那个默认设置。 从法律层面看，私有代码被纳入训练数据，直接触发三个连锁反应：其一，版权归属模糊地带——你的私有代码未经明确授权被用于训练，版权归谁？其二，竞争机密泄露风险——同领域公司的代码被混合训练，模型可能"不经意"学到特定商业逻辑；其三，GDPR合规红线——欧盟开发者若代码涉及用户数据，被未经同意用于训练可能构成数据处理违规。 从行业影响看，这一决策本质上是AI数据军备竞赛的缩影。OpenAI、Anthropic、Google，所有训大模型的公司都在疯狂抢夺高质量代码数据。GitHub作为全球最大代码托管平台，手里握着数万亿美元价值的私有代码金矿，这次决策等于宣布：各位程序员，你们集体罢工了40年的代码主权，正式被一个Opt-Out勾选框出卖了。 HN上最尖锐的评论来自用户vmg12："GitHub不是在给用户选择权，他们是在赌你懒得选。"这句话精准刺穿了这场闹剧的本质。

■ Telnyx PyPI供应链投毒
https://telnyx.com/resources/telnyx-pyth... · https://news.ycombinator.com/item?id=475...

2026年3月，知名VoIP服务商Telnyx官方Python SDK在PyPI上被发现存在恶意代码注入，71名HN用户紧急评论，58 points热度背后是整个Python开发生态的信任危机。Telnyx官方已发布安全通告，承认供应链被攻击。这不是单一事件——这是2026年开源供应链攻击浪潮中最新的一个注脚。

供应链投毒不是什么新鲜事，但这次Telnyx事件有几个极其危险的特征，值得所有工程师警醒。 第一，攻击目标是真实企业签名的官方SDK，而非随机第三方包。攻击者没有伪造一个"telnyx"的山寨包，而是成功渗透了Telnyx官方账号，把恶意代码植入真实的、有合法签名的发布版本。这意味着传统的安全验证方式——检查包名、验证发布者——彻底失效。当官方包都不安全，你还相信什么？ 第二，恶意代码的注入时机精准。攻击者选择在新版本发布后的短时间内完成篡改，利用的是PyPI缺乏强制代码签名验证的漏洞。PyPI目前并不强制所有包启用SCMS（软件物料清单）核查，任何有发布权限的账号都可以在发布后修改元数据而无需重新签名。这是一个结构性缺陷，不是技术bug。 第三，影响面远超Telnyx本身。Telnyx作为企业级VoIP服务商，其SDK被集成进大量生产系统。这次投毒可能影响的不是几个个人项目，而是真实的商业通信系统——呼叫中心、客服平台、SMS网关——这些系统的安全漏洞意味着攻击者可以窃听商业通信、篡改通话路由、窃取认证令牌。 从更大的视角看，2025-2026年，开源供应链攻击已经进化到第四代：第一代是随机恶意包，第二代是仿冒知名包，第三代是混入合法依赖，第四代就是现在这样——拿下官方账号，篡改真实发布。这次Telnyx事件是第四代攻击的典型案例，标志着开源信任模型已经出现系统性危机。 HN上用户ramimac的评论点出了关键："我们整个行业建立在'相信开源包'这个假设上，但这个假设从来没有被严肃验证过。"

#!/bin/bash
# verify_dependencies.sh - 校验hash防止供应链篡改
for pkg in $(pip freeze); do
