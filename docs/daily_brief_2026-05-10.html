一、文档代理的致命软肋：最会写漂亮废话的机器，正在悄悄污染你的知识资产

原始信号：arXiv 论文 LLMs corrupt your documents when you delegate 与 HN 讨论
来源：https://arxiv.org/abs/2604.15597
HN 讨论：https://news.ycombinator.com/item?id=48073246

背景溯源：
过去两年，办公 AI 的卖点一直是“把脏活交给模型”。先是总结会议、润色邮件、扩写提案，再到今天的“代理式编辑”：用户给一个目标，模型自己打开文档、改结构、替换段落、更新表格、顺手把语气调得像咨询公司 PPT。问题在于，文档不是聊天窗口。聊天错了可以补一句，文档错了会进入合同、论文、财报、制度库、产品需求、法律意见书，最后变成组织的长期记忆。LLM 在这里最危险的不是不会写，而是太会写。它能用流畅语言把事实篡改包成高级感，把格式破坏藏在排版正常的外衣里，把一次小改动扩散成全局语义漂移。真正的事故往往不是页面崩了，而是大家过了两周才发现某个定义、数字、引用、责任边界已经被改坏。

完整分析：
这条情报的核心不是“LLM 有幻觉”这种老掉牙废话，而是“委托式编辑”暴露了 Agent 产品的根基问题：模型生成的是自然语言，但企业需要的是可审计操作。传统文档编辑有一个很硬的事实：每一次改动都应该能被定位、对比、回滚、归责。人类编辑会留下修订痕迹，程序化编辑可以留下 AST diff，数据库迁移会留下 migration 文件。但很多 AI 文档代理的默认路径，是把文档塞进上下文，再吐出一个“更好版本”。这在演示里很丝滑，在生产里就是把审计链条一刀砍断。

更毒的是，办公文档天然是混合结构：段落、样式、表格、脚注、引用、评论、权限、版本、附件、嵌入对象全都挤在一起。模型看起来像在改文字，实际上是在碰一个结构化系统。只要它无法稳定识别“只改第 3 节第二段，不碰定义表，不改编号，不删除评论”，它就不是生产工具，只是一个穿着西装的全文重写器。今天企业愿意为 AI 办公付钱，不是因为它能写一段漂亮总结，而是因为它能进入法律、咨询、审计、研究、政企流程这些高价值场景。偏偏这些场景最怕隐性错误。

实战可用的架构原则应该很明确：文档代理不能直接改全文，必须做成“计划、补丁、验证、回滚”的事务模型。模型负责提出变更意图，执行器负责以结构化 API 应用局部补丁，验证器负责比较语义和格式约束，用户最后批准。换句话说，LLM 应该是副驾驶，不是拿着生产数据库 root 权限的实习生。

代码/配置演示：生产级文档代理的最小护栏

目标：只允许 Agent 修改指定段落，并输出可验证补丁，而不是整篇重写。

policy.yaml
agent:
mode: patch_only
allowed_operations:
- replace_paragraph
- insert_after_paragraph
- update_table_cell
forbidden_operations:
- rewrite_document
- delete_comments
- change_heading_numbering
- modify_citations
approval_required:
- any_numeric_change
- legal_terms_change
- table_schema_change

validation:
require_diff_summary: true
require_style_preserved: true
max_changed_paragraphs: 5
numeric_change_requires_source: true
citation_change_requires_user_approval: true

伪代码：
function editDocument(request, document):
plan = llm.createEditPlan(request, document.outline)
assert plan.operations.all(op in policy.allowed_operations)

patches = []
for operation in plan.operations:
target = document.locate(operation.selector)
before = target.snapshot()
after = llm.rewriteLocalBlock(operation.instruction, before)
patch = structuredDiff(before, after)
validatePatch(patch, policy.validation)
patches.append(patch)

preview = renderReviewView(document, patches)
if user.approves(preview):
document.applyTransaction(patches)
auditLog.write(request, plan, patches, user.id)
else:
return cancelled

这个方案不酷，但能活。Agent 行业现在最大的问题就是迷恋“自动完成一切”的戏剧效果，忽略“每一步都能被审计”的工程底线。真正能卖进企业的，不是会写诗的模型，而是会留下证据链的模型。

Amanda Askell 则从对齐角度强调，安全研究不能只盯着“不要做坏事”，还要回答模型应该成为什么。原文链接：https://x.com/AmandaAskell/status/2052928572810256748
这两条合起来看很刺眼：一边是 AI 公司要把模型塞进所有高价值工作流，一边是模型人格和行为边界还在摸石头。企业客户不是慈善家，他们买的是效率，不是帮大厂做社会实验。

实战优劣势分析：
优势一，结构化补丁让 AI 编辑从黑盒魔法变成可审计流水线，能进法律、咨询、金融、政企这些高单价场景。优势二，局部编辑显著降低上下文污染和格式破坏，失败半径小。优势三，审计日志天然服务合规、回滚和责任归属，采购部门能听懂。
劣势也很硬。第一，开发成本高，必须对 Word、PDF、HTML、Markdown、内部知识库分别适配结构化解析器。第二，用户体验没有“点一下全文变好”那么爽，演示不够性感。第三，模型需要更强的定位、计划和自检能力，否则补丁系统会变成另一层错误包装纸。

Dolan's 锐评：
文档代理现在像一个会背麦肯锡黑话的熊孩子，跑得快，手也快，但你不盯着它，它就把合同里的责任主体改成空气。AI 办公的下半场不是谁的总结更丝滑，而是谁敢把改动拆成事务、证据、回滚和责任链。不能做 diff 的 Agent，都是在组织知识库里裸奔。别跟我讲“智能涌现”，先把第 4.2 条别改丢。

二、Anthropic 狂飙与裁员潮同框：AI 不是效率工具，是资本开支换轨的手术刀

原始信号：Latent Space 日报 Anthropic growing 10x/year, everyone else laying off >10%
来源：https://www.latent.space/p/ainews-anthropic-growing-10xyear

背景溯源：
科技行业过去十年最稳定的剧本，是“招人、融资、扩张、讲 TAM、再招人”。疫情后这个剧本被高利率撕开，到了生成式 AI 时代，剧本变成更残酷的一版：AI 原生公司用 ARR 增速讲神话，传统科技公司用裁员和自动化向资本市场交投名状。Latent Space 把 Anthropic 的高增长叙事和“其他公司裁员超过 10%”并置，不是标题党，而是产业资本迁移的截面。钱没有消失，只是从人头预算流向模型订阅、API、推理账单、GPU 集群和自动化平台。以前公司买 SaaS 是给员工提效，现在公司买 AI 是为了证明同样收入可以少养一批人。

完整分析：
这件事最值得拆的不是 Anthropic 到底是不是 10x，而是 10x 背后的收入质量。AI 公司 ARR 口径可能混着长期合约、云抵扣、试点预算、渠道承诺和战略客户预付款。看起来像企业级需求爆炸，实际上可能有一部分是大客户买期权：今天先锁模型能力，明天再决定是不是真替换工作流。模型公司的毛利也不像传统 SaaS 那么漂亮，推理成本、训练成本、芯片折旧、数据采购、安全团队、企业支持全都在吞利润。传统 SaaS 卖的是软件边际成本趋近于零，AI 原生公司卖的是“每次用户用得越多，账单越热”。这就是为什么算力效率、缓存、路由、小模型、MoE、蒸馏会变成董事会话题，不是研究员的玩具。

但需求侧也是真的。企业客户正在把 AI 放进客服、销售、代码、文档、法务、数据分析、运营监控。只要一个工作流能被模型吞掉 30% 的重复劳动，就足以触发 CFO 的条件反射：少招人，慢补人，砍外包，压咨询费。裁员不一定是 AI 单独造成的，但 AI 给了管理层一个体面说法：不是我无情，是组织杠杆升级。翻译成人话，就是用模型采购替代一部分人力预算。

这里还有一个更狠的结构变化：AI 头部公司正在从“模型供应商”变成“组织操作系统供应商”。Claude 不是只卖聊天，它卖企业知识入口、代码协作、文档代理、客服自动化和安全品牌。OpenAI、Google、Anthropic、微软都在争夺同一件事：谁能坐在企业员工每天工作的中间层，谁就能收取新时代办公税。裁员潮释放的不只是成本，也是工作流重构机会。

大佬们的新感悟：
Swyx / Latent Space 把这个对照摆出来，价值在于没有把 AI 增长写成童话，而是放进裁员、预算、组织效率的同一个资产负债表里。原文链接：https://www.latent.space/p/ainews-anthropic-growing-10xyear
Amanda Askell 的正向对齐观点也必须放进商业语境：企业愿意把更多工作交给模型，前提是模型不只是“不杀人放火”，还要诚实、稳定、可解释、知道什么时候该闭嘴。原文链接：https://x.com/AmandaAskell/status/2052928572810256748

实战优劣势分析：
优势一，AI 原生工具真的能吞掉大量重复任务，尤其是客服、内部问答、代码样板、文档初稿、数据清洗。优势二，头部模型公司安全品牌强，能加速企业采购。优势三，裁员和预算冻结反而逼组织寻找自动化替代，AI 公司拿到顺风。
劣势同样扎心。第一，推理成本会持续咬毛利，收入高增长不等于利润高质量。第二，企业工作流集成慢，试点到生产有一道“安全、权限、审计、责任”的铁门。第三，AI 替代叙事会引发组织阻力和监管压力，尤其当白领岗位被系统性挤压时，政策讨论不会一直装睡。

Dolan's 锐评：
Anthropic 的增长故事不是“AI 改变世界”的鸡汤，而是资本市场在重新定价白领劳动。以前 SaaS 帮员工多干活，现在 Agent 帮老板少雇人。谁还在拿“提升协作体验”当主菜，谁就没看懂钱往哪流。真正的牌桌上，模型公司卖铲子，云厂商卖电，企业 CFO 卖掉一部分岗位换利润率。温柔一点叫组织效率，难听一点叫白领杠杆再抵押。

三、MoE 与多模态泛化：大模型从炫肌肉进入省电省钱的苦日子

原始信号：UniPool: A Globally Shared Expert Pool for Mixture-of-Experts
来源：https://arxiv.org/abs/2605.06665v1
原始信号：Are We Making Progress in Multimodal Domain Generalization? A Comprehensive Benchmark Study
来源：https://arxiv.org/abs/2605.06643v1

背景溯源：
过去的大模型竞赛像健身房互相晒围度：参数更大、上下文更长、榜单更高、demo 更花。可进入 2026 后，产业问题变成了两个字：账单。训练要钱，推理要钱，低延迟要钱，多租户服务要钱，企业 SLA 也要钱。MoE 的商业诱惑就在这里：总参数可以很大，但每次只激活一部分专家，用更少计算换更强能力。UniPool 提出的 globally shared expert pool，本质是在问一个更现实的问题：专家能不能跨层、跨任务、跨场景复用，而不是每个模块各自养一堆小山头，最后参数碎片化、路由拥堵、负迁移一地鸡毛。

完整分析：
MoE 的最大价值不是论文指标，而是单位推理成本。如果模型公司每多卖一美元 API，就要烧掉五毛算力，那估值再性感也会被毛利率打回人间。共享专家池试图提高参数利用率，让不同任务从同一组专家里动态取能力。听起来像云计算里的资源池化：不要每个业务线都自己买服务器，而是把能力池化后按需调度。问题是，模型专家不是无状态容器。专家学到的是分布、模式和任务偏好，硬共享可能造成路由竞争、专家过热、负迁移和训练不稳定。它能不能从 paper idea 变成生产方案，关键看三件事：路由是否稳定，专家是否可解释，服务是否可控。

多模态泛化 benchmark 则从另一个角度泼冷水。多模态模型最容易在 demo 里装神：看图答题、识别场景、解释图表、辅助机器人，全都能在精心样本上起飞。但真实世界不是 benchmark 的摄影棚。医疗影像换设备、自动驾驶换天气、机器人换厨房、工业质检换光照，模型就可能从“智能”退化成“猜得很自信”。跨域泛化才是多模态落地的硬门槛。企业不关心你在某个榜单多 1.2 分，它关心模型出了训练分布后会不会把异常当正常，把危险当背景。

把 MoE 和多模态泛化放在一起看，今天的大模型行业正在从两个方向被拧螺丝：成本侧要求更高参数效率，可靠性侧要求更强分布外能力。只有会省钱但不稳，是玩具；只有很稳但贵到没人用，是实验室神像。能活下来的模型公司，必须同时回答 CFO 和安全负责人：每次调用多少钱，失败边界在哪里，证据链怎么留。

实战优劣势分析：
MoE 共享专家池的优势，是把模型能力做成更像基础设施的资源池，降低重复参数和推理浪费；对算力受限的团队尤其有吸引力。多模态泛化评测的优势，是逼团队承认真实世界比榜单肮脏，提前发现部署雷区。两者结合，可以让模型从“跑分资产”转成“运营资产”。
劣势也不客气。MoE 的工程复杂度高，路由、负载均衡、缓存、容错、监控都比 dense 模型麻烦。共享专家池如果没有解释性，排障会很痛苦。多模态泛化 benchmark 则容易被再次刷榜，真正有效的评测必须绑定业务真实数据和高风险失败样本，不能只靠公开数据集自嗨。

Dolan's 锐评：
大模型行业终于从“我参数大我有理”的青春期，进入“你毛利率多少”的中年危机。MoE 是省钱的算盘，多模态泛化是少翻车的保险。未来赢家不是榜单上最会摆拍的模型，而是能在脏数据、低预算、高并发、审计压力里继续干活的模型。AI 公司别再把每次发布会都搞成法术展示了，CFO 的 Excel 比你的 demo 更冷血。
